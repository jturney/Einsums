# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Einsums is a C++20 tensor algebra library that uses compile-time contraction pattern analysis to select optimal operations: BLAS calls, a packed-GEMM backend, or generic algorithms.
A few components rely on C++23 features through the CXX23 compatibility module, which supplies C++20 fallbacks.
It uses a modular CMake build system with Python bindings generated by the vendored apiary codegen tool and a deferred-execution ComputeGraph with optimization passes.
The CUDA and HIP GPU backends and the MPI distributed computing support are works in progress.

## Build Commands

IMPORTANT: activate the conda environment BEFORE the first configure of any fresh worktree or build directory.
A configure without it caches Apple's SDK Accelerate BLAS (whose f2c-style `sdot` returns double and silently corrupts float BLAS-1 results) and fails on OpenMP.
If a configure ran without the env, purge with `cmake -U '*BLAS*' -U '*LAPACK*'` and reconfigure.

```bash
conda activate einsums-dev

# Configure (out-of-source required; always use the directory named `build`)
cmake -S . -B build -GNinja -DCMAKE_BUILD_TYPE=RelWithDebInfo

# Common option additions
#   -DEINSUMS_BUILD_PYTHON=ON     # apiary codegen tool + bindings + python tests (one switch)

# Build everything / a single target
cmake --build build
cmake --build build --target Einsum1_test

# Run all tests / a single test / a Catch2 case directly
ctest --test-dir build
ctest --test-dir build -R "Einsum1"
./build/libs/Einsums/TensorAlgebra/tests/unit/Einsum1_test "[test-case-name]" --einsums:debug:no-attach-debugger
```

Run test binaries interactively with `--einsums:debug:no-attach-debugger`.
Never run two builds or a build plus ctest concurrently against one build directory; verify with `ps aux | grep -E 'ninja|clang'` that a stopped build's compilers actually exited before starting the next.
Header-compile tests invoke the compiler at ctest time; under heavy system load they time out spuriously, so check `uptime` before believing mass COMPILE_ONLY failures and rerun with `ctest --rerun-failed`.

### Local CI legs (Docker)

`./devtools/docker/run-ci-leg.sh <leg>` runs a CI leg in a persistent container (gcc/clang x openblas/mkl, tsan, asan, windows-cross).
The container bind-mounts the repo at CREATION time: when switching worktrees, `docker rm -f einsums-ci-local` first, or the leg silently tests the original tree.
The windows-cross leg cross-compiles the whole tree with clang-cl (compile+link validation only; the GitHub windows runner executes tests).

## Architecture

### Module System

All modules live under `libs/Einsums/<ModuleName>/` with the standard layout:
```
libs/Einsums/<Name>/
  CMakeLists.txt              # calls einsums_add_module()
  include/Einsums/<Name>/     # public headers
  src/                        # implementation files
  tests/unit/                 # unit tests (C++ and test_*_python.py)
  tests/performance/          # benchmarks
```

The dependency-ordered module list lives in `libs/Einsums/CMakeLists.txt`; new modules are created via `libs/create_module_skeleton.py`.
`CMAKE_MODULE_PATH` additions inside a module directory are directory-scoped; export with `set(CMAKE_MODULE_PATH "${CMAKE_MODULE_PATH}" PARENT_SCOPE)` so later sibling modules can `include()` them.

### Key Module Dependency Chain

`TensorAlgebra` -> `LinearAlgebra` -> `BLAS` -> `BLASBase` -> `BLASVendor`

`TensorAlgebra::einsum()` dispatches through `Backends/Dispatch.hpp`, selecting BLAS specializations > packed-GEMM (`try_packed_gemm`) > generic algorithm.
The outer-product fast path requires each operand's indices to be contiguous within C (the #257/#283 gate); interleaved targets route to the generic algorithm.

### ComputeGraph (deferred execution)

ComputeGraph is the PREFERRED mechanism for new computational code: capture once, optimize with passes, replay many times.
The optimization passes, automatic GPU offload, and distributed execution all operate on captured graphs; eager execution gets none of them.
Eager remains the reference semantics: it is the oracle the differential tests compare against, and simple numpy-style examples stay eager with no passes.

`cg::einsum`/`cg::gemm`/... capture into a `Graph` under a `CaptureGuard`, then `graph.execute()` replays; outside capture the same calls run eagerly.
String-spec einsums lower through `StringDispatch.hpp::string_einsum`, whose fast-path cascade assumes each index letter appears once per operand; repeated-letter (diagonal) specs route to the repeat-aware generic loop.
Semantics enforced at this layer: zero-extent operands still apply the C prefactor exactly once; an output overlapping an input throws `std::invalid_argument` unless the aliased operand's index list is identical to C's (pure elementwise in-place); A and B may always share a tensor.
Overlap rejection triggers only on PROVABLE overlap (both regions contiguous or identical base pointers), because disjoint column-major slices of one parent interleave in memory without sharing elements.
`dispatch::last_dispatch_route()` (thread-local) names the kernel route the most recent `string_einsum` call took; tests use it to assert fast paths fire instead of silently falling back to the generic loop.
Optimization passes are in `src/Passes/`; apply with `graph.apply(default_pass_manager())`.
Returning-form ops (`dot(A,B)`, `syev_eig`, `svd`, ...) throw during capture by design; use the pointer-writer forms.

### Tensor Types

- `Tensor<T, Rank>` - dense tensor, always column-major at construction
- `TensorView<T, Rank>` - non-owning view; `data()` returns the offset-adjusted pointer
- `RuntimeTensor<T>` / `RuntimeTensorView<T>` - runtime-rank variants (the Python-facing types)
- `BlockTensor`, `TiledTensor`, `TiledRuntimeTensor` - structured variants
- `T.data()` raw pointer, `T.dim(i)`, `T.stride(i)` (strides in elements)

### Index System

Indices are compile-time structs created with `MAKE_INDEX(x)`; index letters drive contraction analysis.
Repeated letters within one operand mean diagonal access; a letter in only one input and absent from C is summed (trace).

### SIMD Runtime Dispatch

The SIMD module (`libs/Einsums/SIMD/`) provides compile-time intrinsics wrappers plus a runtime feature ladder (`RuntimeFeatures.hpp`): psABI rungs Baseline/V2/V3/V4, `cpu_features()` (CPUID + XCR0 OS-state gated), `selected_arch()`, and the `EINSUMS_SIMD_ARCH` env override (lower-only, clamps).
Kernel TUs compile once per rung via `einsums_add_simd_dispatch_sources()` (see `SIMD/cmake/Einsums_AddSIMDDispatch.cmake`); HPTT is the reference consumer, dispatching at plan creation.
`EINSUMS_WITH_SIMD_DISPATCH=OFF`, non-x86 targets, or the `EINSUMS_SIMD_NATIVE_ARCH`/`EINSUMS_SIMD_TARGET_CPU` pins collapse to a single native TU.
`einsums_add_simd_rung_tests()` re-registers a test per rung; unsupported rungs report ctest "Skipped" via the `simd_rung_guard` launcher.
MSVC ABI note: classes with out-of-line members need `EINSUMS_EXPORT` on the explicit instantiation DEFINITIONS, not just on the class template or extern declarations.

### Test System

Tests use Catch2 v3 and pytest.
```cmake
einsums_add_executable(MyTest_test INTERNAL_FLAGS SOURCES MyTest.cpp NOINSTALL)
einsums_add_unit_test("Modules.ModuleName" MyTest)        # -> Tests.Unit.Modules.<Module>.MyTest
einsums_add_python_unit_test("Modules.<M>" Name SCRIPT test_name_python.py)
```

- `einsums_add_executable` already links libEinsums; do NOT add `target_link_libraries` for Einsums modules in test CMakeLists (duplicate symbols).
- Python tests live per-module under `libs/Einsums/<Module>/tests/unit/test_*_python.py`, never in the top-level `tests/`; run manually with `PYTHONPATH=build/lib python -m pytest <file>` (bare `pytest libs` collects nothing).
- Hypothesis differential fuzzers (`test_hyp_*_python.py`) compare graph/eager results against numpy oracles; when adding an einsum feature or spec class, extend the GENERATOR draws (repeated letters, traces, zero extents, aliasing, dtypes, complex prefactors are already drawn) rather than only adding fixed cases.
- Known-bug convention: tag failing cases `[!shouldfail]` with a comment naming the bug; when the fix lands the tag turns the run red as a reminder to drop it.
  Capabilities that do not exist yet get placeholder tests that assert the current rejection and describe the replacement test.
- `einsums.testing` provides `ALL_DTYPES`/`REAL_DTYPES`/`COMPLEX_DTYPES`, `tolerance_for`, `assert_close`; prefer parametrizing over `ALL_DTYPES` (float64-only graph tests have hidden real bugs).
- Zero-extent inputs are valid: empty BLAS/LAPACK calls are quick-return no-ops (never let `lwork`-style workspace formulas hit zero) and empty contractions still scale C by its prefactor.

### CMake Options

Key options (`-D<option>=ON/OFF`):
- `EINSUMS_WITH_TESTS` / `EINSUMS_WITH_TESTS_UNIT` / `EINSUMS_WITH_TESTS_BENCHMARKS`
- `EINSUMS_BUILD_PYTHON` - codegen tool + bindings + python tests in one switch
- `EINSUMS_WITH_SIMD_DISPATCH` - multi-rung SIMD kernels (default ON)

Build options map to preprocessor defines: `EINSUMS_WITH_FOO=ON` -> `EINSUMS_HAVE_FOO` via `<Einsums/Config.hpp>`.

### Python Bindings

Bindings are generated by the vendored apiary tool from `APIARY_*` annotations in headers (see `APIARY_EXPOSE` / `APIARY_MODULE` / `APIARY_INSTANTIATE_AS` / `APIARY_INSTANTIATE_BOOLS` in `ComputeGraph/Operations.hpp`).
To expose a new operand type (e.g. `RuntimeTensorView<T>`), add instantiation lines; the codegen folds them into the existing dtype-dispatched python entry and regenerates the `.pyi` stubs.
Missing `EINSUMS_EXPORT` on a symbol breaks `_core.so` dlopen even when C++ tests link fine.
Binding functions that print must `std::cout.flush()` for pytest capfd.

### External Dependencies

Required: BLAS/LAPACK, HDF5, OpenMP (all from the einsums-dev conda env).
Auto-fetched if missing: fmt (12+), Catch2, spdlog (built from source).
Optional: MPI (Open MPI or MPICH).
Each dependency has a `cmake/Einsums_Setup<Dep>.cmake`.

### Distributed Computing (MPI, work in progress)

The `Comm` module (`libs/Einsums/Comm/`) provides MPI communication with a mock backend for serial builds:
**ProcessGrid** (2D Pr x Pc grid with row/col sub-communicators), **DistributionDescriptor** (per-dimension grid axis assignment), **Collectives** (blocking and non-blocking).

ComputeGraph passes for automatic distribution: `DistributionPlanning`, `Materialization`, `InputSlicing`, `SUMMAExpansion`, `CommunicationInsertion`, `CommunicationScheduling`.

```bash
mpirun -np 4 ./build/libs/Einsums/ComputeGraph/tests/unit/DistributedIntegration_test
```
